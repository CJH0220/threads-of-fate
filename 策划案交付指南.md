# 游戏策划案交付指南

> 适用项目：《命运的织线》  
> 适用对象：负责游戏策划、系统设计、文案设计、内容交付的人  
> 目标：把创意整理成研发、美术、UI、测试都能执行的交付物

---

## 1. 核心原则

策划案不是“灵感记录”，而是让团队知道：

1. **要做什么**
2. **为什么做**
3. **规则是什么**
4. **有哪些数据**
5. **怎么判断做对了**
6. **哪些东西暂时不做**

一个合格的策划交付物，应该能让研发人员在不反复追问的情况下，拆出任务、估算工作量、开始实现。

---

## 2. 不同文档的用途

不要把所有内容都塞进一个 Markdown。不同阶段、不同对象应该使用不同格式。

| 交付物 | 推荐格式 | 用途 | 面向对象 |
|--------|----------|------|----------|
| 项目概念介绍 | PPT | 说明游戏卖点、风格、核心体验 | 投资人、团队、外部沟通 |
| 总体策划案 / GDD | Word / Notion / Markdown | 游戏整体设计总纲 | 全团队 |
| 系统策划案 | Markdown / Word | 说明某个系统的规则和逻辑 | 研发、测试、策划 |
| 数值表 / 配置表 | Excel / Google Sheets / CSV | 存放可配置数据 | 研发、数值策划 |
| 剧情大纲 | Word / Markdown | 说明主线、支线、阶段节奏 | 文案、剧情、研发 |
| NPC设定表 | Excel / Sheets / Markdown表格 | 管理角色信息、关系、业线 | 文案、系统、研发 |
| 事件表 | Excel / Sheets / CSV | 管理事件触发、条件、结果 | 研发、文案、测试 |
| UI流程图 | Figma / FigJam / Miro / draw.io | 说明界面流转和交互 | UI、研发 |
| 世界观设定集 | Word / Notion / Markdown | 统一风格、名词、价值观 | 文案、美术、音频 |
| 测试验收清单 | Markdown / Sheets | 判断功能是否完成 | QA、研发、策划 |

## 4. Markdown、Word、PPT、Excel分别什么时候用

### 4.1 Markdown

适合：
- 系统规则
- 技术沟通
- 版本管理
- 研发可读文档
- 验收标准

优点：
- 适合 Git 管理
- 结构清晰
- 容易被 AI 和研发工具读取

不适合：
- 大量图片排版
- 对外展示
- 大型表格维护

---

### 4.2 Word / Notion / Confluence

适合：
- 总体策划案
- 世界观设定
- 剧情文档
- 文案规范

优点：
- 阅读体验好
- 适合长文说明
- 非技术人员容易接受

不适合：
- 高频数据表维护
- 精确配置数据

---

### 4.3 PPT

适合：
- 立项介绍
- 卖点展示
- 给团队快速讲清方向
- 对外融资/汇报

PPT应该回答：
1. 这是什么游戏？
2. 玩家为什么会想玩？
3. 核心循环是什么？
4. 差异化在哪里？
5. 现在要做的版本范围是什么？

PPT不适合写详细系统规则。

---

### 4.4 Excel / Google Sheets

适合：
- NPC属性表
- 技能表
- 事件表
- 资源消耗表
- 业线节点表
- 缘线初始关系表

研发最喜欢这种表，因为它可以直接转成 CSV / JSON / ScriptableObject / Godot Resource。

---

### 4.5 Figma / Miro / draw.io

适合：
- UI流程图
- 系统流程图
- 玩家操作路径
- 状态机
- 事件触发流程

文字说不清的流程，就画图。

## 5. 给研发看的系统策划模板

每个系统策划都建议用这个结构。

```markdown
# 系统名

## 1. 系统目标
这个系统解决什么问题？给玩家什么体验？

## 2. 玩家能看到什么
玩家在界面上看到哪些信息？能做哪些操作？

## 3. 核心规则
用明确规则写清楚系统如何运作。

## 4. 数据结构
列出需要存储的数据字段。

## 5. 输入与输出
输入：玩家操作、时间推进、NPC状态  
输出：资源变化、事件触发、UI变化

## 6. 公式与数值
列出计算公式、默认值、范围、上限下限。

## 7. 状态与流程
状态有哪些？如何切换？

## 8. 边界情况
异常、失败、极端值如何处理？

## 9. 示例
给1-3个完整例子。

## 10. 验收标准
研发做完后，如何判断正确？
```

## 6. 系统策划排查清单

写完一个系统策划后，逐项检查。

| 检查项 | 是否完成 |
|--------|----------|
| 系统目标是否清楚？ | ☐ |
| 玩家能看到的信息是否列出？ | ☐ |
| 玩家能做的操作是否列出？ | ☐ |
| 系统内部状态是否列出？ | ☐ |
| 数据字段是否明确？ | ☐ |
| 公式是否明确？ | ☐ |
| 默认值、最大值、最小值是否明确？ | ☐ |
| 成功/失败条件是否明确？ | ☐ |
| 边界情况是否写清？ | ☐ |
| 是否有完整例子？ | ☐ |
| 是否有验收标准？ | ☐ |
| 是否说明哪些内容暂时不做？ | ☐ |

## 7. 内容策划模板

内容策划包括剧情、NPC、地点、事件、文案。

```markdown
# 内容模块名

## 1. 内容目标
这个内容在游戏中承担什么功能？

## 2. 出现时机
在哪一周、哪个阶段、什么条件下出现？

## 3. 涉及角色
哪些NPC参与？

## 4. 涉及地点
在哪些地点发生？

## 5. 玩家可干预点
玩家可以用什么方式影响？

## 6. 分支结果
可能走向哪些结果？

## 7. 系统影响
会影响哪些数值？香火、阴德、阳德、缘线、业线？

## 8. 文案风格
语气、尺度、禁忌、关键词。

## 9. 验收标准
这个内容做完后怎么验收？
```

---

## 8. NPC策划模板

```markdown
# NPC姓名

## 1. 基础信息
姓名、年龄、职业、外貌、常驻地点。

## 2. 角色定位
这个角色在故事和系统中有什么作用？

## 3. 背景故事
简洁说明过去经历。

## 4. 核心愿望
他/她最想要什么？

## 5. 内外矛盾
内在矛盾是什么？外部阻力是什么？

## 6. 初始属性
心智、信仰、体魄、魅力等。

## 7. 性格模型
善良、激进、感性、理性、好奇心等。

## 8. 初始缘线
与其他NPC的关系。

## 9. 主业线
5-10个节点。

## 10. 支线
2-3条支线。

## 11. 结局可能性
好结局、坏结局、中性结局。

## 12. AI使用等级
S/A/B/C，决定是否进入完整Agent系统。
```

## 9. 事件表模板

事件表建议用 Excel / Sheets，而不是只写 Markdown。

| 字段 | 说明 |
|------|------|
| event_id | 事件唯一ID |
| event_name | 事件名称 |
| week_phase | 所属周阶段 |
| location | 地点 |
| participants | 参与NPC |
| trigger_condition | 触发条件 |
| player_intervention | 玩家可干预方式 |
| coin_check | 是否需要命运硬币判定 |
| success_result | 成功结果 |
| fail_result | 失败结果 |
| resource_change | 香火/阴德/阳德变化 |
| fate_line_change | 业线变化 |
| relationship_change | 缘线变化 |
| ai_text_needed | 是否需要AI生成文本 |
| safety_note | 内容尺度备注 |

## 10. 文案策划应该怎么写

文案策划不是只写漂亮句子，而是要让文本可控、可替换、可审核。

### 10.1 文案分类

| 类型 | 示例 | 建议格式 |
|------|------|----------|
| 固定UI文案 | 按钮、提示、资源名 | 表格 |
| 系统提示 | “神力不足” | 表格 |
| 剧情对白 | NPC对话 | Word / Markdown |
| AI生成提示词 | 给LLM的上下文和约束 | Markdown |
| 事件描述 | 事件开头/结果 | 表格+模板 |
| 结局文案 | 主结局+后日谈 | Markdown / Word |

### 10.2 文案表字段

| 字段 | 说明 |
|------|------|
| text_id | 文案ID |
| scene | 使用场景 |
| speaker | 说话人 |
| text | 文案内容 |
| tone | 语气 |
| condition | 出现条件 |
| forbidden | 禁止内容 |
| ai_variant | 是否允许AI改写 |

## 11. AI原生游戏额外需要的策划文档

《命运的织线》是 AI 原生游戏，因此还需要普通游戏没有的文档。

| 文档 | 内容 |
|------|------|
| AI生成边界文档 | 哪些内容AI能写，哪些不能写 |
| NPC Memory规范 | 角色记忆如何存储、压缩、读取 |
| Prompt模板文档 | 给AI的系统提示、角色提示、事件提示 |
| 内容安全规则 | 暴力、性、邪教、未成年内容如何处理 |
| AI成本预算表 | 每天/每时段最多调用多少次 |
| AI失败兜底方案 | AI超时、违规、生成质量差时怎么办 |

## 14. 最重要的写作习惯

写策划时，尽量把“感觉”转成“规则”。

| 不够好 | 更适合研发 |
|--------|------------|
| NPC会慢慢喜欢对方 | 缘线强度每次正向互动 +5，达到60触发暧昧事件 |
| 玩家可以影响命运 | 玩家可消耗神力改变硬币概率或数量 |
| 邪教会扩散 | 邪教传播度每周结算，达到30/60/90触发不同事件池 |
| 角色有记忆 | 每个S级NPC保留50条事件记忆，超过后摘要压缩 |
| 结局由AI生成 | 系统先判定主结局，再把NPC业线、缘线、地点状态作为AI输入生成后日谈 |

策划案不是越长越好，而是越明确越好。

## 15. 你的工作重点

你现在作为游戏策划，最应该优先完成：

1. **把创意整理成系统规则**
2. **把角色整理成表格数据**
3. **把剧情整理成事件池**
4. **把AI自由度限制在可控边界内**
5. **给研发明确输入、输出、数据字段、验收标准**

只要做到这些，研发就能把你的策划拆成任务并逐步实现。
