# UI 规范：本地化与文案基线

> 文档状态：初稿  
> 更新日期：2026-06-12  
> 所属阶段：P0 全局规范  
> 引擎方向：Godot  
> 美术方向：简约像素风

---

## 1. 用途

本文定义《命运的织线》UI 文案、本地化、文本长度和排版扩展规则。所有 UI 文案必须能被后续替换为多语言文本，避免写死在脚本或图片中。

---

## 2. 基本原则

- UI 文案使用文本 key 管理，不直接写死在逻辑脚本中。
- 关键按钮、状态、错误、提示文案必须可本地化。
- 不把关键文字烘焙进图片。
- 文本容器预留约 40% 长度扩展空间。
- 文本密集区域必须支持换行或滚动。
- 中文文案保持克制、清楚，不用过长诗化句子承载操作信息。

---

## 3. 文本 key 命名建议

使用层级式 key：

```text
ui.<screen>.<element>.<state>
ui.<component>.<element>.<state>
```

示例：

| 文案 | Key 示例 |
|---|---|
| 继续游戏 | `ui.main_menu.continue` |
| 新游戏 | `ui.main_menu.new_game` |
| 加载游戏 | `ui.main_menu.load_game` |
| 推动时间 | `ui.hud.advance_time` |
| 神力不足 | `ui.toast.divine_power_not_enough` |
| 删除存档 | `ui.confirm.delete_save.confirm` |
| 当前没有事件 | `ui.state.no_events.title` |

---

## 4. 字符长度建议

| 元素 | 中文建议长度 | 处理方式 |
|---|---|---|
| 主按钮 | 2–8 字 | 超出时按钮加宽或换短词 |
| 危险确认按钮 | 4–10 字 | 必须明确后果 |
| Tab 标签 | 2–5 字 | 避免长句 |
| Toast | 4–18 字 | 超出时允许两行 |
| 空状态标题 | 4–12 字 | 简短说明状态 |
| 空状态说明 | 8–32 字 | 可换行 |
| 事件正文 | 不限 | 使用滚动或分页 |
| 结局文本 | 不限 | 使用滚动或分段 |

---

## 5. 排版扩展规则

- 按钮宽度按中文文本预留 40% 扩展。
- 设置项左侧标签宽度固定，右侧控件可伸缩。
- 列表卡片标题最多两行，超过后显示省略号并提供详情。
- Toast 最多两行，超长文本应改为弹窗或详情面板。
- 事件文本、结局文本、说明文本不得依赖单行显示。
- 小字号文本不能承载关键操作说明。

---

## 6. 文案风格

### 6.1 操作文案

操作文案必须直接说明动作：

| 不推荐 | 推荐 |
|---|---|
| 确定 | 开始编织命运 |
| 是 | 删除存档 |
| 返回 | 返回主菜单 |
| 继续 | 继续推动时间 |

### 6.2 错误文案

错误文案必须说明：

1. 发生了什么。
2. 玩家现在可以做什么。

示例：

```text
读取存档失败。
存档可能损坏或版本不兼容，请选择其他存档。
```

### 6.3 危险文案

危险文案必须明确但不剧透：

```text
仍有危险征兆尚未查看。
如果继续推动时间，可能错过干预机会。
```

---

## 7. 动态数值格式

动态文案统一使用占位符：

```text
ui.resource.incense_delta = "香火 {value}"
ui.time.day_period = "第 {day} 天 · {period}"
ui.save.updated_at = "最近保存：{time}"
```

规则：

- 不把数字和单位拆成无法重排的多个硬编码 Label。
- 正负数显示由格式化层统一处理。
- 时间、日期、数字格式后续应集中管理。

---

## 8. 不可本地化内容

以下内容可以不进入本地化表：

- 调试 UI。
- 仅开发期可见占位符。
- 资源内部 ID。
- 美术文件名。

但只要玩家可见，就必须进入本地化流程。

---

## 9. Godot 实现建议

建议 UI 层只引用 key：

```text
LocalizationService.tr(key, params)
```

文本数据建议集中在表格或资源文件中：

```text
res://data/localization/zh_cn.csv
res://data/localization/en_us.csv
```

---

## 10. 验收标准

- [ ] P0 UI 的按钮、标题、错误、提示文案都有 key 规划。
- [ ] 按钮和 Tab 预留文本扩展空间。
- [ ] 事件正文和结局文本支持换行或滚动。
- [ ] Toast 超长时不会遮挡关键 UI。
- [ ] 危险操作按钮文案明确说明后果。
- [ ] 所有玩家可见文本后续可从本地化表替换。
