## 1. 已定稿的架构决策

### 1.1 总体技术路线

| 决策 | 结论 |
|------|------|
| 游戏引擎 | **Godot + Python 运行时** |
| Godot 职责 | 表现层：画面渲染、音效、输入、动画 |
| Python 职责 | 逻辑层：游戏规则、事件结算、NPC 行为、LLM 调用、存档 |

### 1.2 前后端通信

| 决策 | 结论 |
|------|------|
| 框架 | **FastAPI**（同时支持 REST + WebSocket） |
| REST 用途 | 简单同步操作：初始化、存档/读档、查询 |
| WebSocket 用途 | 双向实时通信：时间推进结算、LLM 流式文本、关键警报推送 |
| 数据格式 | **JSON**，统一消息外壳 |

**统一消息格式：**

```python
class Message(BaseModel):
    type: str            # 消息类型
    payload: dict        # 实际数据
    timestamp: float     # 时间戳
    request_id: str | None = None  # 请求追踪

class ApiResponse(BaseModel):
    success: bool
    data: Any | None = None
    error: str | None = None
    changed_fields: list[str] | None = None  # 增量模式
```

**WebSocket 消息类型：** `advance_time`, `settlement`, `npc_dialogue`, `alert`, `llm_stream`

### 1.4 数据模型层

所有数据结构使用 **Pydantic** 定义，作为前后端的数据合同。前端人员可以直接阅读 `models/` 目录了解 API 返回的数据结构。

### 1.5 配置与状态的分离

| 类型 | 加载时机 | 是否可变 | 内容 |
|------|----------|----------|------|
| **静态配置** | Python 启动时加载一次 | 只读 | 地点配置、事件模板、干预技能、结局判定公式、本地化文本、周阶段参数 |
| **动态状态** | 每次新游戏创建 GameSession 实例 | 运行时持续变化 | NPC 状态、缘线、业线进度、资源值、世界状态 |

**CSV 是蓝图，GameSession 是实例。** 存档保存的是 GameSession 的快照，不修改 CSV。

### 1.6 游戏主循环状态机

```
[启动] → [初始化] → [观察] ←──────────────────┐
                       ↓                       │
                    [干预]                      │
                       ↓                       │
                    [时间推进]                  │
                       ↓                       │
              Python 结算（事件+AI+LLM）        │
                       ↓                       │
                    [结果展示] ──→ 下一时段 ──→─┘
                       ↓
              [周结算]（每7天）
                       ↓
              [结局判定]（第60天 / 强硬不幸）
                       ↓
                    [终局]
```

**核心 API 端点（预留扩展空间）：**

| 端点 | 协议 | 触发 | 返回 |
|------|------|------|------|
| `POST /new-game` | REST | 开始新游戏 | 全量初始状态 |
| `POST /intervene` | REST | 玩家执行干预 | 干预结果确认 |
| `POST /advance-time` | WS | 玩家结束时段 | 增量结算结果 |
| `GET /state` | REST | 任意时刻查询 | 当前完整状态 |
| `POST /save/{slot}` | REST | 保存到指定槽位 | — |
| `POST /load/{slot}` | REST | 从指定槽位加载 | 全量状态 |
| `GET /saves` | REST | 列出所有存档 | 存档元信息列表 |
| `DELETE /save/{slot}` | REST | 删除存档 | — |

### 1.7 数据流动粒度

| 场景 | 策略 |
|------|------|
| 新游戏初始化 | **全量返回**所有 NPC、缘线、资源、世界状态 |
| 读档 | **全量返回**完整 GameSession |
| 时段结算 | **增量返回**仅变化的部分（NPC、缘线、资源、事件列表） |

### 1.8 NPC Agent 设计

每个 NPC 拥有**独立的 Agent 实例 + 独立的记忆 + 独立的 LLM 对话上下文**：

```python
class NpcAgent:
    static: NpcStatic       # 静态层（来自配置，不变）
    dynamic: NpcDynamic     # 动态层（每时段更新）
    memory: AgentMemory     # 记忆层（事件链、关键记忆、情感印象）
    llm: LLMClient          # 独立的 LLM 对话历史
```

**记忆存储：** 当前方案为**内存 + JSON 序列化**，抽象接口预留 SQLite 扩展能力。

**NPC 分级：**
- S 级（6人）：完整 Agent，独立记忆 + LLM
- A 级（4人）：半完整 Agent，关键记忆 + 事件触发时 LLM
- B 级（3人）：规则驱动，无 LLM
- C 级（群体/轻量）：固定反应，不进入 Agent Memory

### 1.9 LLM 接入

| 决策 | 结论 |
|------|------|
| 调用方式 | **异步任务队列**，不阻塞游戏主流程 |
| NPC 调用 | 每个 S/A 级 NPC **独立调用**，从各自记忆和性格出发 |
| 后端切换 | **抽象接口**，运行时可选 DeepSeek API 或本地模型 |
| 降级策略 | LLM 不可用时，NPC 使用**规则兜底行为**（按日常日程表行动） |

```python
class LLMClient(ABC):
    async def chat(self, messages: list[dict], **kwargs) -> str: ...
    async def chat_stream(self, messages: list[dict]) -> AsyncIterator[str]: ...

class DeepSeekClient(LLMClient): ...
class LocalModelClient(LLMClient): ...
```

### 1.10 事件系统

采用**责任链模式**：

```
事件触发 → 命运硬币判定（规则引擎） → 影响分发到子系统：
  ├── 资源引擎（香火/神力）
  ├── 缘线引擎（关系变化）
  ├── 业线引擎（进度推进）
  ├── NPC 状态引擎（位置/情绪）
  └── 世界状态（邪教值等）

→ LLM 润色文本（异步） → 返回 Godot
```

**事件数据结构：** `EventTrigger`（触发条件）→ `GameEvent`（事件定义）→ `EventOutcome`（结果）

### 1.11 缘线图存储

基于现有 `src/backend/graph.py` 的 `RelationshipGraph`（邻接表 + 边字典双索引），迁移进新结构后继续使用。支持有向边、强度(0-100)、活跃值(0-100)、5 种关系类型（红/金/蓝/灰/黑）。

### 1.14 错误处理与容错

| 场景 | 处理 |
|------|------|
| Python 进程未启动 | Godot 显示"正在连接服务器..."，自动重试 |
| WebSocket 断线 | 自动重连，期间缓存玩家操作 |
| LLM API 超时 | 该 NPC 使用规则兜底行为（日常日程表） |
| LLM 返回异常 | 丢弃输出，使用本地模板 |
| 存档文件损坏 | 校验 + 提示用户 |
| FastAPI 内部异常 | 全局异常中间件捕获，返回 ApiResponse(success=false) |

**核心原则：任何单一失败不应让游戏整体崩溃。**

## 2. 待讨论问题

以下问题已在框架讨论中提出，但需要与其他团队成员进一步讨论后决定。

### Q1：LLM 决策范围 —— NPC 行为控制 vs 游戏状态修改

**当前矛盾：**

- 用户需求：云端 LLM 需要控制 NPC 的行动
- 策划案（`AI生成边界.md`）：规则决定结果，AI 只包装表达；AI 不能改变资源数值、NPC 状态、缘线强度、业线节点、结局路线

**待讨论方向：**

是否采用「决策层」与「执行层」分离？

| 层 | 谁决定 | 例子 |
|----|--------|------|
| NPC 要干什么（决策层） | LLM（基于性格+记忆+处境） | "林潮音决定去海边找陈远舟" |
| 干了之后发生什么（执行层） | 规则引擎 | 触发事件 → 结算资源/缘线变化 |

即：LLM 决定 NPC 的**意图**，规则引擎决定**后果**。

**影响系统：** `ai/npc_agent/`, `engine/event/`, `AI生成边界.md`

### Q2：业线结局判定时机

业线达到 100% 后的结局判定：

- **方案 A：** 立即判定结局类型（好/坏/中性），60 天终局确认
- **方案 B：** 延迟至 60 天终局统一判定

**影响系统：** `engine/karma/`, `engine/event/`
