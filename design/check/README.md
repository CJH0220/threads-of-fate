# check 检查与验收目录

> 文档状态：整理版  
> 更新日期：2026-06-12  
> 用途：策划检查表、研发交付验收清单和版本范围清单

---

## 1. 目录职责

`check/` 用于存放项目进入实现前和实现过程中的检查清单。它不承载新设计，而是用于确认已有设计是否完整、一致、可交付。

本目录回答的问题是：

- 当前版本到底做什么，不做什么？
- 策划文档是否命名统一？
- UI、系统、内容、数据是否能互相对上？
- 研发拿到文档后是否可以拆任务？
- 测试是否能根据文档验收？

---

## 2. 当前检查清单

| 文件 | 用途 | 建议使用时机 |
|---|---|---|
| `MVP范围检查.md` | 确认首版 MVP 范围是否收敛 | 每次准备进入实现前 |
| `命名一致性检查.md` | 检查角色、地点、资源、系统 ID 是否统一 | 每次批量修改策划或数据后 |
| `交付检查清单.md` | 检查 UI / system / content / data 是否可以交给研发 | 每次阶段性交付前 |
| `完整策划交付总览.md` | 汇总当前完整策划案、权威文件和研发入口 | 正式交付策划案时 |

---

## 3. 检查状态标记

检查项统一使用：

```text
- [ ] 未完成
- [x] 已完成
- [ ] 待讨论：说明原因
```

如果检查项暂不适用于当前版本，保留该项并标注：

```text
- [ ] 暂不适用：原因
```

---

## 4. 使用原则

- 检查清单只记录是否满足，不在这里展开新设计。
- 如果发现缺口，应回到对应目录补文档：`UI/`、`system/`、`content/`、`data/`。
- 如果发现命名冲突，应优先更新权威索引：`design/README.md`。
- 如果检查项影响 MVP 实现，应标记为 P0。
- 如果只是后续体验增强，应标记为 P1 / P2。
