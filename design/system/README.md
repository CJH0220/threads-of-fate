## 1. 目录职责

`system/` 用于存放面向研发和测试的系统策划文档。

本目录回答的问题是：

- 系统目标是什么？
- 玩家能看到什么？
- 数据从哪里来？
- 规则如何结算？
- 哪些数值可调？
- 有哪些边界情况？
- 如何验证系统完成？

内容剧情写在 `content/`，界面规格写在 `UI/`，系统规则写在 `system/`。

## 3. 建议系统文档结构

每个系统文档建议包含：

1. 概述
2. 玩家可见信息
3. 核心规则
4. 数据结构
5. 输入与输出
6. 公式或判定规则
7. 与其他系统的依赖
8. 边界情况
9. 示例流程
10. 调参项
11. 验收标准

## 6. 实现原则

当前项目以 Godot 和低成本 MVP 为目标，系统设计应遵循：

- 规则驱动优先，AI 不直接决定核心结果。
- 数据本地化优先，先使用 JSON / CSV / Godot Resource。
- 所有关键数值可调，不写死在剧情文本中。
- UI 只展示状态，不拥有核心游戏状态。
- 事件、NPC、资源、结局应能被测试用例验证。
- 复杂关系图、自然语言查询、大规模 Agent 推演放到 P2 或更后。
